# Sisters Directory — UFS Mangalore Province

Flutter + Firebase app with three parts:
1. **Login** (`lib/screens/login/login_screen.dart`)
2. **Admin dashboard** (`lib/screens/admin/`) — matches your dashboard mockup
3. **Sister "Complete your profile"** (`lib/screens/user/complete_profile_screen.dart`)

Routing is automatic: after login, `AuthGate` checks the signed-in user's role
in Firestore (`users/{uid}.role`) and sends **admin** → dashboard, **sister** → profile screen.

---

## 1. Prerequisites (install once)

```bash
# Flutter SDK (if not already installed)
git clone https://github.com/flutter/flutter.git -b stable ~/flutter
export PATH="$PATH:~/flutter/bin"
flutter doctor

# Node.js (needed for Firebase CLI) + Firebase CLI + FlutterFire CLI
npm install -g firebase-tools
dart pub global activate flutterfire_cli
```

## 2. Create the project (skip if you're dropping these files into an existing app)

```bash
flutter create sisters_directory
cd sisters_directory
```

Then copy every file from this delivery into the matching path shown in the
**File structure** section below (they overwrite/extend `lib/`, add
`firestore.rules`, `firestore.indexes.json`, `firebase.json`, and update `pubspec.yaml`).

## 3. Install dependencies

```bash
flutter pub get
```

## 4. Connect Firebase

```bash
firebase login
firebase projects:create sisters-directory   # or use an existing project ID

# from the project root (sisters_directory/)
flutterfire configure --project=sisters-directory
```

`flutterfire configure` will:
- ask which platforms to support (choose Android, iOS, Web as needed)
- **overwrite** `lib/firebase_options.dart` with your real API keys
- register Android/iOS apps and drop `google-services.json` /
  `GoogleService-Info.plist` into the right native folders automatically

## 5. Enable Firebase products (in the Firebase console, or via CLI)

```bash
# Enable Email/Password sign-in
# Console: Authentication -> Sign-in method -> Email/Password -> Enable

# Deploy Firestore rules + indexes
firebase deploy --only firestore:rules,firestore:indexes
```

## 6. Seed your first admin user

Create the user in **Authentication → Add user** (Firebase console) with an
email/password, copy the generated UID, then add a matching Firestore doc:

```bash
# Using the Firebase console: Firestore -> Start collection -> "users"
# Document ID = the UID you just copied
# Fields: role (string) = "admin"
```

Or via a quick script (`tool/seed_admin.dart`, included) — see file for usage.

## 7. Run the app

```bash
flutter run                 # pick a connected device/emulator
flutter run -d chrome       # run in the browser
flutter run -d macos        # desktop, if configured
```

## 8. Build for release

```bash
flutter build apk --release          # Android
flutter build ios --release          # iOS (needs Xcode + signing)
flutter build web --release          # Web
```

## Firestore collections this expects

| Collection    | Purpose                                   |
|---------------|--------------------------------------------|
| `users`       | `{uid}` → `{ role: "admin" \| "sister" }`  |
| `sisters`     | full sister profile records                |
| `communities` | community/house records                    |
| `documents`   | uploaded documents metadata                 |
| `photos`      | gallery photo metadata                      |
| `obituaries`  | obituary records                            |
| `activities`  | dashboard activity log (auto-appended)      |

## Notes

- The dashboard is fully responsive: sidebar collapses into a `Drawer` under
  1000px width, and every card row reflows into a column on narrow screens.
- Charts use `fl_chart` (donut for Community Distribution, bar for Age
  Distribution); the calendar uses `table_calendar`.
- Every stat card, chart, and list on the dashboard is wired to a live
  Firestore `Stream` — no dummy data, it reflects your actual collections.
- Wire up the five tiles on the "Complete your profile" screen
  (`Personal Information`, `Education`, `Formation`, `Religious Life`,
  `Documents & Photo`) to your existing user-module forms via the `onTap`
  callbacks — they're left as `TODO` since those forms already exist in
  your user module.
